## Context

当前项目使用 NestJS + LangChain 构建智能对话系统，已具备以下能力：
- 基础对话功能（通过 [`runChain()`](src/ai/ai.service.ts:86) 实现）
- 消息历史记录（使用 `InMemoryChatMessageHistory`）
- 工具调用能力（查询用户信息、创建定时任务）
- 定时任务调度系统（基于 `@nestjs/schedule`）

用户需要在对话中获得项目管理支持，包括进度跟踪、风险识别和提醒功能。

## Goals / Non-Goals

**Goals:**
- 在 `runChain()` 函数中增强 AI 的 PMO 管理能力
- AI 能够自动识别用户消息中的项目进度更新
- AI 能够识别潜在的项目风险并给出建议
- AI 可以创建定时或延时任务来提醒用户项目相关事项
- 保持与现有工具调用机制的兼容性

**Non-Goals:**
- 不创建独立的项目管理服务（PMO 逻辑内嵌在 AI 对话中）
- 不持久化项目状态（依赖现有消息历史记录）
- 不改变现有的定时任务调度架构

## Decisions

### 1. PMO 能力实现方式

**决策**: 通过增强 system prompt 实现 PMO 能力，而非创建新工具

**理由**:
- 项目进度识别和风险识别是认知型任务，适合由 LLM 直接处理
- 减少工具调用复杂度，提升响应速度
- 与现有 `create_scheduled_task` 工具无缝集成

**替代方案考虑**:
- 方案 A: 创建独立的 `track_progress` 和 `identify_risk` 工具
  - 缺点：增加工具调用复杂度，需要额外的 schema 定义
- 方案 B: 创建独立的 PMO 服务
  - 缺点：过度设计，当前需求简单，内嵌实现更轻量

### 2. System Prompt 增强

**决策**: 在现有 system prompt 基础上增加 PMO 相关指令

**增强内容**:
- 项目进度识别规则（完成、进行中、阻塞、延期等状态）
- 风险识别规则（资源不足、时间紧张、依赖风险等）
- 提醒任务创建引导

### 3. 项目状态存储

**决策**: 复用现有的 `chatHistories` Map 存储项目状态上下文

**理由**:
- 避免引入新的存储层
- 项目状态与对话历史天然关联
- 保持架构简洁

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| AI 可能过度识别风险，造成用户困扰 | 在 prompt 中明确风险识别的阈值和条件 |
| 项目状态未持久化，重启后丢失 | 当前为 MVP 版本，后续可扩展为持久化存储 |
| System prompt 过长可能影响模型表现 | 精简 prompt 内容，只保留核心规则 |
| 与现有工具调用逻辑冲突 | 充分测试，确保工具调用优先级正确 |
